# Agent Cowork v1.0 迭代计划(草案 · 待讨论)

> 状态:讨论稿。基线 = v0.1.0(`releases/v0.1.0/`,本地 Windows 桌面 Agent:Tauri 2 + Node host(SEA) + React,Kimi API)。
> 目标版本:**v1.0 —— 一个你敢每天用、能干复杂活、能发给别人用、不被单一模型锁死的本地办公智能体。**

---

## 一、为什么要分阶段(而不是四个方向一起上)

你选的四条主线(打磨 / 扩能力 / 可分发 / 多模型)都要,但它们有**依赖与风险次序**:

- 能力(子代理、多模态)如果建在不稳的基座上,bug 会成倍放大 → 先加固基座。
- 真沙箱隔离、连接器是"放开手脚干活"的安全前提 → 在大幅扩能力的同时/之前补。
- 可分发(签名、自动更新、跨平台)应在功能稳定后再做,否则每次大改都要重签重测。
- 多模型抽象可以**并行**推进(纯后端解耦),不阻塞 UI。

所以排成 P0(基座)→ P1(能力)→ P2(安全+连接器)→ P3(多模型)→ P4(分发),其中 P3 可与 P1/P2 并行。

## 二、贯穿全程的原则

1. **每次运行都有交代**:不空回复、不静默终止、错误可读、可取消、可"继续"。
2. **写操作永远先批准**:能力越强,审批与审计越要清晰(批量批准/记住本会话/可回滚)。
3. **诚实优先**:做不到的(如本地网络隔离)就明说并用测试锁定,不假装。
4. **可复现**:每个里程碑都打 tag + 出离线快照(git bundle)+ 安装包归档。
5. **小步验证**:每个特性都要有单测/集成测 + 一次真实模型端到端实跑截图。

## 三、分阶段路线图

### P0 · 基座加固(为后续大改铺路,1 个迭代)
- **测试稳定性**:全量用例在 `--test-isolation=process` 下绿;修掉只在无隔离时挂的 conversations/exec-child 等;CI 脚本(本地一键)。
- **工程健壮**:规避"挂载盘写文件被截断"这类环境坑(构建/写盘加校验);release 流程脚本化(tag→bundle→签名→归档)。
- **提供商抽象 seam**:把"调用模型"收敛到一个 provider 接口(为 P3 多模型铺路,先不接新模型)。
- 验收:`test:host`/`test:ui` 全绿且可复现;一条命令产出"可发布快照"。

### P1 · 能力深化:让它能干复杂的活(2~3 个迭代,价值核心)
1. **计划—执行—自检闭环 + 可见 Todo**
   - 计划模式升级:研究→提计划→批准→执行→自检→收尾,全程一个可勾选的 Todo 列表在 UI 实时更新。
   - 验收:一个"多文件重构/多步整理"任务能走完闭环,Todo 勾选齐全,产物自检通过。
2. **子代理并行**
   - 主代理把可并行的子任务派发给子代理(已有 subagent 雏形),并行执行 + 汇总;每个子代理独立审批与上下文预算。
   - 验收:一个"同时分析 3 个文件夹并汇总"的任务并行跑通,耗时明显低于串行。
3. **多模态(读图 / 出图 / 文档产物)**
   - 读图:已有多模态雏形,补齐"贴图→理解→据图操作"。
   - 出物:接入文档技能(docx/xlsx/pptx/pdf)+ 图表(已有 chart/mermaid 内联)+ 简单图像生成(可选)。
   - 验收:"把这张表格截图整理成 Excel"和"根据要点生成一页 Word"端到端可用,产物能一键打开。
4. **审批 / 进度 / 错误 UX 打磨**(让"每天敢用"落地)
   - 批量批准、记住本会话同类操作、更清晰的工具卡片(参数/结果/失败原因)、明确的取消与"继续"。
   - 验收:一次含 5+ 工具调用 + 多次审批的任务,操作顺滑、状态一目了然。

### P2 · 安全 + 连接器:放开手脚但有边界(1~2 个迭代)
1. **真沙箱网络隔离(补齐"那五分")**
   - 检测到 Docker/WSL 时,默认走 VM 后端 `--network=none` 真隔离;没有则回退本地并**明确提示"本地不隔离网络"**(已有诚实性测试)。
   - 验收:装了 WSL/Docker 的机器上,联网命令在隔离模式下被真实阻断(集成测试)。
2. **连接器生态(MCP)**
   - 一键安装/管理 MCP server + 精选注册表;常用连接器(GitHub / Notion / 邮箱 / 数据库)+ OAuth 流程;每连接器细粒度权限与审批。
   - 验收:从面板一键装一个连接器并在对话里成功调用,权限可控、可撤销。

### P3 · 多模型 / 不绑定 Kimi(可与 P1/P2 并行)
- 在 P0 的 provider 抽象上接入多提供商(Kimi / OpenAI / Claude / 本地 OpenAI 兼容端点);模型路由 + 失败降级链;用量/成本面板;每会话可切模型;BYO-key 体验统一。
- 验收:同一任务可在 2 个提供商间切换跑通;成本面板显示 token/费用;某家挂了能自动降级。

### P4 · 可分发 / 上线(功能稳定后,1~2 个迭代)
- **真 CA 代码签名**(OV/EV)消除 SmartScreen「未知发布者」;签名流程已就绪,缺真证书。
- **自动更新**(Tauri updater + 签名校验 + 更新源)。
- **跨平台**:macOS(.dmg / .app,sidecar 改 mac 二进制)、Linux(.AppImage/.deb);sidecar 解析与路径已为多平台预留。
- **安装包瘦身** + 首次运行引导 + 可选崩溃上报(opt-in、脱敏)。
- 验收:Windows 真签名安装包无 SmartScreen;应用内可检测并完成一次更新;至少再跑通 1 个平台。

## 四、跨阶段横切项
- 文档:每个里程碑更新图文上手指南 + 变更说明(CHANGELOG)。
- 性能:启动时间、内存占用、host 拉起时延的基线与回归看护。
- 可观测:本地 metrics/审计已具备,补一个"运行历史/用量"视图。
- 安全复审:每阶段做一次轻量审计(沿用已有 strict-audit / phoenix 等效复扫)。

## 五、需要你拍板的开放问题(影响排期与成本)
1. **跨平台范围**:v1.0 是否一定要 macOS/Linux?(工作量大;可只锁 Windows,跨平台放 v1.1)
2. **真证书**:CA 代码签名证书(OV/EV)需要花钱 + 主体资质,你这边能提供吗?否则继续自签 + 文档说明。
3. **图像生成**:多模态的"出图"是否要(需要可用的图像生成 API/模型)?还是先只做"出文档/图表"。
4. **遥测**:是否允许 opt-in 的匿名崩溃/用量上报?(默认关闭、可拒)
5. **节奏**:每个迭代你期望多大颗粒(一次给一个完整可装的版本,还是按特性小步多次)?

## 六、建议的第一刀(P0 + P1 的最小切片)
若认可方向,建议先做:**①P0 测试稳定性 & release 脚本化 ②P1-1 计划闭环 + 可见 Todo**。
这两块价值高、风险低、且为后面所有特性奠基;做完出 v0.2.0 标签 + 安装包给你试。
