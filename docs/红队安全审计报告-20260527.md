## 二、中危详情:SSRF-1

`tools/web-fetch.js` 是面向"研究"任务的显式出站抓取工具,默认 `allowInternal=false` 时用正则封锁内网:

```js
const BLOCKED_HOST_RE = /^(localhost|0\.0\.0\.0|127\.|10\.|192\.168\.|169\.254\.|::1|\[::1\])/i;  // :10
...
if (!allowInternal && BLOCKED_HOST_RE.test(parsed.hostname)) { throw fail(...); }                 // :52
response = await fetchImpl(parsed.href, { signal: controller.signal, redirect: 'follow' });        // :62
```

**绕过点(均可达内网):**

1. **私网段漏挡**:正则缺 `172.16.0.0/12`(172.16–172.31)、IPv6 ULA `fc00::/7`(fd00::)、IPv6 链路本地 `fe80::`。`http://172.20.x.x/` 直接放行。
2. **只校验字面主机名,不校验解析后的 IP**:`BLOCKED_HOST_RE.test(parsed.hostname)` 检的是字符串。攻击者用一个解析到 `127.0.0.1` / `169.254.169.254` / `172.20.x.x` 的域名(DNS rebinding)即可绕过。
3. **跟随重定向但不复核跳转目标**:`redirect:'follow'`,封锁检查只在初始 URL 做一次。外部 URL 返回 `302 → http://169.254.169.254/` 会被自动跟随,目标主机不再校验。
4. **IP 编码未规整**:`http://2130706433/`(=127.0.0.1 的十进制)、八进制/十六进制写法可绕过字面正则。

**可利用链**:恶意网页内容 → prompt injection 诱导 agent 调用 `web.fetch` 抓内网/本机服务 → 响应文本回灌进模型上下文(信息泄露 / 内网探测)。桌面场景无云 IAM,危害低于服务器,但 LAN 内部服务(路由器后台、本机其它端口)探测真实存在。

**修复建议:**
- 解析后**对最终 IP**判定(`dns.lookup` 后比对),而非字面主机名;覆盖全部私网/保留段(含 172.16/12、IPv6 ULA/链路本地、`0.0.0.0/8`、CGNAT 100.64/10)。
- **每一跳重定向都重新校验**(改 `redirect:'manual'`,逐跳解析+判定),或限制最大跳数。
- 规整 IP 编码(用 `net.isIP` / 解析为规范形式再判定)。

## 三、低危 / 设计权衡

### REBIND-1:DNS-rebinding 与 GET 信息泄露(已被大幅缓解)
- Origin 中间件(`http/middleware/common.js:69`)只对 `/api/*` 的 **写方法**(POST/PUT/PATCH/DELETE)强制 `isAllowedOrigin`,**GET 不在校验范围**;且全局无 Host 头反 rebinding 校验(`isLoopbackHostname` 只用于校验 Origin 的 hostname,未用于校验请求 Host)。
- **为何仍是低危**:鉴权走 `Authorization: Bearer`(非 cookie),浏览器不会跨域/重绑后自动携带 token;`requireAuth` 默认开。攻击页即便重绑成功也拿不到有效 token,读不到受保护数据。
- **建议(纵深防御)**:加一个 Host 头白名单中间件(只接受 `127.0.0.1`/`localhost`/`[::1]`/`tauri.localhost`),对所有方法生效。低成本,直接堵死 rebinding。

### CFG-1:绑定地址可被环境变量放开
- `const host = process.env.HOST || '127.0.0.1'`(`main.js:8`)默认安全,但设 `HOST=0.0.0.0` 即对外暴露,无告警/护栏。
- **建议**:非 loopback 绑定时打印醒目告警,或要求显式 `--allow-remote` 才放行。

### CFG-2:`trustIdentityHeaders` 是把双刃剑
- `auth/request-identity.js:32-38`:开启后直接信任客户端 `x-tenant-id`/`x-user-id` 头并置 `authenticated=true` → **任意调用方冒充任意租户**。默认关(`KCW_TRUST_IDENTITY_HEADERS==='true'` 才开),源码注释也承认"仅因 loopback+CORS 才可接受"。
- **风险**:该理由不完整——CORS 拦不住非浏览器客户端;若同时部署到非 loopback 或被 rebinding,即成租户越权。
- **建议**:仅在确有反向代理注入可信头时启用,并在启用时强制要求 Host/来源白名单;文档明确标注危险。

### RCE-1 / CMD-1:命令执行(按设计的能力,非漏洞)
- PowerShell 工具 `powershell.exe -NoProfile -NonInteractive -Command <command>`(`kimi/agent-tools.js:227`)传入完整命令串 = 任意代码执行,但"每条命令都需用户确认"。
- `workspace/command-runner.js` 默认禁用(需 `allowCommands===true`)、`shell:false`、`assertTrustedPath` 钳 cwd、超时+上限缓冲;但**不限制可执行的二进制**。
- **建议**:确保审批 UX 无法被静默自动通过;对 command-runner 增加可选二进制白名单做纵深防御。

## 四、值得肯定的防护(红队未能突破)

- **无 shell 注入**:`cli-runner`/`cli-detect`/`mcp stdio`/`sandbox exec-child`/`command-runner` 全部 `shell:false` + argv 传参;`sandbox-spec.js` 明确"argv 值无法注入 shell 语法"。
- **JWT 实现正确**:`auth/jwt.js` 硬性要求 `alg==='HS256'`(挡 `alg:none` 与 RS256→HS256 混淆)、`timingSafeEqual`+长度校验(防时序)、校验 exp/nbf 带时钟容差。
- **鉴权默认安全**:`requireAuth` 默认开;租户/用户**绝不从客户端头读取**(默认),只由验证过的 JWT/会话覆盖;基于 Bearer token(非 cookie)→ 天然抗 CSRF。
- **CSRF 防护已接线**:Origin 中间件对写操作返回 403;`withJsonBody` 强制 `content-type: application/json`(强制跨域预检)。
- **路径越狱防护健壮**:`path-policy.js` 做 realpath 规范化、create-aware 祖先回溯、Windows 8.3 短名/junction 处理、敏感段/敏感扩展名封锁、escape-root 拒绝。
- **沙箱**:默认断网(`network=false`)、`timeoutMs`、SIGKILL、上限缓冲;结构化白名单守护 VM/server 后端;内置 Python 不放大调用方白名单。
- **DoS 护栏**:请求体 1MB 上限 → 413;输出缓冲有上限。
- **Prompt-injection 检测**:`kimi/safety/untrusted-content.js` 检测"call/run/execute shell"等注入模式。
- **密钥脱敏**:`redactText`/`redactValue` 已接入错误日志、模型韧性日志、recipe 固化(标 `redacted:true`)。

---

## 五、优先级建议

1. **SSRF-1(中)**:按"解析后 IP 判定 + 逐跳重定向复核 + 编码规整"修 `web-fetch.js`。这是唯一一个与配置无关、对已鉴权调用方/被注入 agent 都成立的真实代码缺口,建议优先。
2. **REBIND-1(低,纵深)**:加 Host 头白名单中间件,一次性堵死 rebinding,成本极低。
3. **CFG-1 / CFG-2(低,设计)**:加非 loopback 绑定告警;给 `trustIdentityHeaders` 加强约束与文档警示。
