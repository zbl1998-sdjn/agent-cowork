# v0.3 Implementation Status

> Baseline: `plan/kimi-cowork-latest-product-plan-v0.3.md`  
> Current implementation goal: keep MVP-0 as a local proof of concept, and start MVP-1 with architecture-compatible skeletons.

## Baseline Decision

v0.3 is the source of truth.

- Office Mode is the default product experience.
- Developer Mode is explicit and isolated.
- Kimi is the default model brain.
- OpenAI-compatible adapters are allowed for Developer Mode.
- Local files are processed only after workspace trust.
- High-risk actions must be previewed, approved, logged, and rollback-capable.

## Current Track

### MVP-0 Kept

The existing Node host remains as `apps/host`.

It validates:

- Kimi CLI detection.
- trusted root checks.
- file tree and file reading.
- context bundles.
- file operation preview/apply with explicit no-overwrite protection for write, rename, and move.
- JSONL audit.
- host API operation smoke coverage for file tree, file read, context bundle, preview, apply, blocked overwrite, and JSONL audit.

### MVP-1 Started

The first MVP-1 skeleton now uses v0.3's formal direction:

- `apps/windows-client`: C/Win32 desktop UI with a Kimi-style Office Mode shell.
- `apps/local-agent`: Go Local Agent with policy, file tools, journal, and file operations.
- `services/api`: multi-user API skeleton.
- `services/relay`: device relay session skeleton.
- `services/orchestrator`: task state machine skeleton.
- `services/kimi-gateway`: Kimi/OpenAI-compatible gateway primitives.

## MVP-1 P0 Acceptance

- Local Agent path policy blocks escapes and sensitive paths.
- Local Agent file tools can list/read/hash trusted files.
- File operations support write, rename, move; delete is forbidden.
- Journal records operation events.
- API service exposes health and basic task/workspace route stubs.
- Orchestrator state machine rejects invalid transitions.
- Kimi Gateway has explicit timeouts and bounded retry helpers.
- Windows C client skeleton configures and builds with Visual Studio Community 2026 Developer PowerShell plus Ninja.
- Windows C client now renders a visible front end: sidebar, central Kimi prompt, task templates, trusted workspace picker, file list, artifact/approval preview, and Developer Mode panel.
- Windows C client operation smoke test launches the app with `--workspace`, verifies local file scanning, Generate Plan, Approval, and Developer Mode.
- Windows C client Approval now writes a safe Markdown deliverable under `.KimiCowork/artifacts`, appends `.KimiCowork/audit/audit.jsonl`, and creates a rollback journal under `.KimiCowork/rollback`.
- Windows C client Generate Plan now reads trusted-root TXT / Markdown / CSV snippets into a local content summary before approval.
- Windows C client Generate Plan now includes a minimal safe move preview for the first scanned trusted-root file.
- Windows C client Approval can apply that safe move into `Kimi_Cowork整理/<template>/`, appending both audit and rollback journal entries.

## Local Build Notes

The Windows client needs the Visual Studio developer environment because `cl.exe`
is not loaded in a regular PowerShell session. Verified command:

```powershell
& 'C:\Program Files\Microsoft Visual Studio\18\Community\Common7\Tools\Launch-VsDevShell.ps1' -Arch amd64 -HostArch amd64 -SkipAutomaticLocation
cmake -S apps/windows-client -B build/windows-client-vs -G Ninja
```

Operation smoke command:


generates a plan with local text summaries and a minimal move preview, approves
it, and checks that the Markdown artifact, moved file, audit events, and
rollback entries are written inside the trusted workspace.
